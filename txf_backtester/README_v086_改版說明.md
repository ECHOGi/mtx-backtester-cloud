# v0.8.6 長回測續跑＋新指標效能修正版

## 問題確認

v0.8.5 的多截止日未來情境結果，只有在全部策略情境完成後才會寫入
`st.session_state`、產生ZIP並上傳Google Drive。執行期間的數千筆結果只存在
記憶體中的 `rows` 清單；瀏覽器分頁休眠、網路中斷、Streamlit工作階段重建
或程序中止時，已完成進度會全部消失，所以畫面會回到回測前狀態。

這與JSON檔案大小無直接關係。拆成10組後仍有3,600筆，而且v0.8.5新增指標
原本會對所有策略無條件預算SAR、BIAS、寶塔線與開盤缺口，條件判斷時部分
指標又重新計算，使單筆成本高於v0.8.4。

## 修正內容

### 1. 情境結果檢查點

- 每完成10筆情境結果，即追加寫入JSON Lines檢查點。
- 檢查點識別包含策略內容、資料範圍、截止日、路徑數、seed、成本、部位與
  正二設定，只有完全相同設定才會續跑。
- 畫面或工作階段中斷後，以相同設定再按「開始回測」，平台會讀取已完成列，
  略過已完成的截止日／情境／路徑／seed／策略組合。
- 檢查點尾端即使因中斷留下半行，也只略過該行，不破壞先前結果。
- 提供「忽略同設定的中斷進度，從頭重跑」勾選項；平常不勾選。

### 2. 固定共同路徑seed

所有截止日、六種情境與路徑seed會在開始時先建立完整任務表。續跑不會因略過
已完成工作而改變後續seed，確保續跑結果與一次完整跑完一致。

### 3. 前端進度節流

進度列最多約更新200次，不再每完成一筆就向瀏覽器傳送訊息。3,600或7,200筆
長回測可大幅減少不必要的前端通訊。

### 4. 新指標按需計算

- 未使用SAR出場時，不再為所有策略預先建立SAR欄位。
- BIAS、日K缺口與寶塔線只在策略條件實際引用時才計算。
- 同一策略、同一路徑、同參數的SAR／BIAS／缺口／寶塔線結果會快取一次。
- 同一共同路徑中，指標參數相同的策略變體會共用基礎指標資料。

### 5. 回測邏輯不變

本版沒有修改SAR、BIAS、缺口、寶塔線公式，也沒有修改進出場、部位、正二基準
或排名口徑。使用相同合成資料、相同seed與batch_028A進行60筆對照，v0.8.5與
v0.8.6的24個輸出欄位逐列完全一致。

## 操作方式

1. 部署v0.8.6。
2. 維持原本的batch_028A或batch_028B設定開始回測。
3. 若畫面再次重置，所有選項保持相同，再按一次「開始回測」。
4. 側欄會顯示「發現同設定中斷進度：N筆，將自動續跑」。
5. 不要勾選「忽略同設定的中斷進度，從頭重跑」，除非確定要清除舊進度。

檢查點位於執行主機的 `/tmp/txf_backtester_checkpoints`。一般分頁中斷與工作階段
重建可續跑；若雲端主機整台被重新建立、重新部署或清除暫存磁碟，該本機檢查點
仍可能消失，因此A、B拆批仍保留作為額外保險。
